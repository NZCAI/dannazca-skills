# PRD: Nazca Shared API & MCP Connector (Orchestration Infrastructure)

## 1. Problem Statement
Nazca operates multiple independent software products (e.g., a public-facing WhatsApp Copilot and an internal Trading Agents workflow). While these products are functionally decoupled, they require a shared, scalable infrastructure to handle routing, background execution, and tool orchestration. Crucially, our agentic systems (like LangChain/LangGraph) need a standardized way to trigger n8n workflows (like `TOOL INDICADORES` and `TOOL SNAPSHOTS`) using the Model Context Protocol (MCP).

## 2. Core Product Distinctions (Scope)
The API acts as a generic interoperability layer for distinct applications sharing the same infrastructure:
- **Public Services**: External-facing products like the WhatsApp Copilot. The FastAPI handles secure proxying and webhook ingestion from n8n.
- **Internal Tools**: Products like the Trading Agents workflow. Operated internally, requiring execution of heavy asynchronous scripts.
- **Interoperability**: Products do not inherently talk to each other, but the FastAPI layer ensures they can share the same underlying database schemas and n8n toolsets without tight coupling.

## 3. Key Functional Requirements

### MCP Connector Implementation
- The API must implement the standard MCP spec to expose n8n workflows as executable tools for LLMs.
- Example: Expose the `TOOL INDICADORES` webhook so an agent can pass arguments like `fund_code` (N1, N2, N3) and `company` to fetch financial data directly from the Langchain/LangGraph orchestration layers.

### Decoupled Routing
- **Public Route**: Ingestion endpoints for n8n Webhooks related to the WhatsApp Copilot.
- **Internal Route**: Execution endpoints to trigger the Trading Agents script in the background.
- **Data Pipelines**: Routes to trigger Cron-based Database-to-S3 ingestion via n8n.

### Data Storage
- Strictly pass-through. The API will NOT store state locally or in a database. State management and storage are strictly delegated to the end-services (S3, n8n, etc.).

## 4. Technical Architecture

### Framework & Concurrency
- **Framework**: Python FastAPI acting as a stateless, shared infrastructure module.
- **n8n Interoperability**: FastAPI translates standard JSON API requests (or MCP tool calls) into specific n8n webhook payloads.
- **Concurrency Strategy**:
  - Uses `async def` for I/O bound webhook routing to ensure fast response times (< 200ms) for public services.
  - Uses FastAPI `BackgroundTasks` for internal LLM scripts (like TradingAgents) to prevent hanging the main event loop.

## 5. Open Questions for Next Phase
1. **MCP Tool Definitions**: Should the FastAPI dynamically read the n8n tool schemas, or will we hardcode the Pydantic models for tools like `TOOL INDICADORES` into the FastAPI codebase?
2. **Public vs Internal Separation**: Should we separate the API into two different Docker containers/ports (one public-facing, one internal-only) for tighter security, or use a single instance with strict path-based firewalls?
